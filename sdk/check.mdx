---
title: "check()"
description: "Ask whether an action would be allowed, without running it."
---

```ts
check(action: ActionDescriptor): Promise<Decision>
```

A policy decision without side effects. Same payload as `guard()`, no execution, no `report()` follow-up.

Use it when you want to surface a UI hint ("would this be allowed?"), test a new policy without running anything, or audit a non-signing action.

## Returns

```ts
type Decision = {
  allowed: boolean;
  reason?: string;          // human readable, e.g. "exceeds 50 per-tx USDC"
  reason_code?: string;     // machine readable, e.g. "over_per_tx_cap"
  audit_event_id: string;   // UUID of the audit row written for this check
  cached: boolean;          // true if served from local cacheTTL
  evaluated_at: string;     // ISO timestamp
};
```

A `check()` call writes an audit row so the chronicle records that a decision was requested. No `tx_signature` is attached (since nothing ran).

## Failure-mode behaviour

If `failureMode: "open"` and the backend is unreachable, `check()` resolves with a synthetic permissive decision:

```ts
{
  allowed: true,
  reason: "sdk_unreachable_failopen",
  reason_code: "sdk_failopen",
  audit_event_id: "00000000-0000-0000-0000-000000000000",
  cached: false,
  evaluated_at: "<now>",
}
```

If `failureMode: "closed"`, `check()` throws `AltheiaConnectionError`.

## Example

```ts
const decision = await altheia.check({
  type: "transfer",
  asset: "USDC",
  amount: 1200,
  target: recipient,
});

if (!decision.allowed) {
  notifyOperator(`would deny: ${decision.reason}`);
}
```

## When to prefer `check()` over `guard()`

- You want to show the operator "would this be allowed?" before they confirm.
- You're testing a new policy without risk of accidentally running anything.
- Your action doesn't sign on-chain (you're only inferring intent or doing a dry-run).

For anything that signs, [`guard()`](/sdk/guard) is the right call. It gives you the audit row with the signature attached.
