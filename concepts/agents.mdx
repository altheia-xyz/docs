Every Altheia integration has two actors and three keypairs. Get this picture right and the rest of the docs are just details.

## The two roles

| Role | Who | Where it signs | What it signs |
| --- | --- | --- | --- |
| **Operator** | A human, you | Phantom in the dashboard | Lifecycle: register, pause, unpause, revoke. ~4 transactions per agent. |
| **Agent** | Your code | Session keypair in your script | Every real action: swaps, transfers, program invocations. |

The operator's Phantom wallet **never signs an agent action**. Your script's session key **never signs a lifecycle action**. The split is enforced by the program: only the operator's PDA can call pause/revoke, only the session key is scoped to the agent's policy.

<Note>
  This is the trust shift that makes Altheia worth running. Even if the Altheia backend is compromised tomorrow, the attacker has no way to sign as your agent. The session-key secret was generated in your browser and never sent over the wire.
</Note>

## The three keypairs

```
Operator wallet  (Phantom)         → signs lifecycle events
   └─ OperatorAccount PDA          → derived from Phantom address

Agent identity   (PDA)             → not a keypair, just an address
   └─ AgentAccount PDA             → derived from operator PDA + 32-byte agent ID
   └─ stores policy hash, status, model commitment

Session keypair  (your env)        → signs every agent action
   └─ generated in your browser
   └─ pubkey is whitelisted on the Swig session-key scope
   └─ funded directly via faucet/transfer to its pubkey
```

## Agent identity, on-chain

Each agent is an `AgentAccount` PDA:

```
seeds   = ["agent", operator_pda, agent_id_bytes]
program = AkKx54ZmuP17r1sXsKr7mxe3dXJ5RMqsSH2zf8QGZ39C  // devnet
```

`agentPda` is the canonical reference everywhere: SDK config, dashboard URL, audit rows, on-chain memos. Treat it the way you'd treat a user ID.

What the on-chain account stores:

- **Policy commitment.** sha256 of the canonical policy JSON. The full policy lives in Altheia's database, but the hash on Solana is the source of truth for "what policy was active when this transaction signed."
- **Model commitment.** sha256 of the model identifier you registered with. Lets auditors prove the agent's model didn't silently change.
- **Status.** One of `active`, `paused`, `revoked`, `archived`.
- **Session key pubkey.** The keypair allowed to sign for this agent.

## Lifecycle

| Status | What it means | How to get out |
| --- | --- | --- |
| `active` | Agent can sign actions that pass policy. | Pause or revoke. |
| `paused` | SDK denies every action with `agent_paused`. Session key intact. | Unpause. |
| `revoked` | SDK denies with `agent_revoked`. Swig scope is destroyed. | **No path back.** Register a new agent. |
| `archived` | Hidden from default views. Already revoked. | Unarchive (cosmetic). |

Pause is for "something is off, investigating." Revoke is for "this agent is dead to me." Once revoked, even if someone copies the session key secret, it can't sign anything on-chain because Swig no longer recognizes it.

## Why PDAs and not random IDs

Every state transition lands on-chain under the agent's PDA. Policy commitments, status flips, audit roots, all anchored. An auditor with a block explorer can prove "this agent had this policy at slot N" without trusting Altheia's database or asking us for anything.
