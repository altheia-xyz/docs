A **policy** is a per-agent JSON document that defines what the agent may do. Altheia enforces it in two places: the SDK pre-flight (off-chain, fast) and Swig's session-key scope (on-chain, the floor).

## Shape

```json
{
  "asset_caps": {
    "BRjpCHtyQLNCo8gqRUr8jtdAj5AjPYQaoqbvcZiHok1k": {
      "max_per_tx": 50,
      "max_per_day": 500
    },
    "So11111111111111111111111111111111111111112": {
      "max_per_tx": 0.5,
      "max_per_day": 2
    }
  },
  "allowed_programs": [
    "whirLbMiicVdio4qvUfM5KAg6Ct8VwpYzGff3uctyCc"
  ],
  "blocked_destinations": [
    "BadActorAddress11111111111111111111111111111"
  ],
  "max_actions_per_minute": 10
}
```

The example uses devnet mints: `BRjpCHty...` is devnet USDC, `So11111...` is wrapped SOL. On mainnet (future) the USDC mint is different. Match the chain you're on.

## Fields

| Field | What it controls | Enforced on-chain? |
| --- | --- | --- |
| `asset_caps[mint].max_per_tx` | Max units per single transaction. | Yes. Swig `TokenLimit`. |
| `asset_caps[mint].max_per_day` | Max units in a rolling 24h window. | Yes. Swig `TokenRecurringLimit`. |
| `allowed_programs` | Program IDs the agent may invoke. | Yes. Swig `ProgramScope`. |
| `blocked_destinations` | Addresses the agent may never send to. | No. Off-chain only. |
| `max_actions_per_minute` | Rate limit on guarded calls. | No. Off-chain only. |
| `expires_at` | Auto-revoke at this ISO timestamp. | No. Off-chain trigger. |

Amounts are in **human units**: `50` means 50 USDC, `0.5` means 0.5 SOL. The SDK and dashboard convert to base units when calling the chain.

## Two layers, one truth

```
your agent code
  │
  └── altheia.guard(action, fn)
        │
        ├── 1) /sdk/agent_check  → off-chain pre-flight
        │     │
        │     ├── allowed → run fn()
        │     └── denied  → throw PolicyDeniedError (fn never runs)
        │
        └── fn() signs the tx with the session keypair
              │
              └── Swig session-key scope enforces the same caps on-chain
                    └── over-cap tx fails at signing, no fee burned
```

The pre-flight is fast feedback. The on-chain scope is the one you trust. If the SDK is misconfigured or the backend is unreachable, Swig still refuses to sign anything outside its programmed limits.

This is why `failureMode: "open"` is the SDK default. The on-chain floor catches over-cap actions even when the off-chain check is unavailable. For agents that move real money, set `failureMode: "closed"` so denials halt instead of swinging.

## Updating a policy

Edit on the agent detail page. You sign one transaction. The on-chain policy commitment updates, the Swig scope updates, and the new policy takes effect on the next `guard()` call. Transactions already in-flight are not retroactively blocked.
